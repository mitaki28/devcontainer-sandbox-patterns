# 基本コンポーネント: mitm-proxy — 粗粒度・暗黙的な操作許可

本章は本リポジトリの 2 つの基本コンポーネント ([02-design.md](./02-design.md) §1.4) のうち、**粗粒度・暗黙的な操作許可 (ホスト × HTTP メソッド × パスで通信を許可し、通信成立 = 操作成立とみなす)** を担う `lib/mitm-proxy` の位置付けと機能を述べる。もう一方 (細粒度・明示的な操作許可) は前章 [04-mcp-proxy.md](./04-mcp-proxy.md) の mcp-proxy。実装詳細・内部構造は [`lib/mitm-proxy/README.md`](../lib/mitm-proxy/) に集約する。

## 1. 章のスコープ

`lib/mitm-proxy` は、作業コンテナ側から外部への HTTP(S) の外向き通信を **mitmproxy で TLS 終端** し、その中身を見て **ホスト × HTTP メソッド × パス** の単位で許可 / 拒否するプロキシである。これは [02-design.md](./02-design.md) §1.4 で導入した 2 つの基本コンポーネントのうち、**粗粒度・暗黙的な操作許可** を担う側にあたる (もう一方は [04-mcp-proxy.md](./04-mcp-proxy.md) の mcp-proxy)。

一般的な粗粒度の外向き通信制御 (Squid の SNI ACL、iptables の domain/IP 単位等) はホスト単位までしか絞れず、ホストを 1 つ許可すれば読み取り / 書き込み問わず通る。本プロキシはホスト × HTTP メソッド × パスまで絞れるため **読み取りのみ許可を書ける** ことが特徴で、[02-design.md](./02-design.md) §3.1 の運用原則 (粗粒度の mitm-proxy は読み取りのみを主とし、書き込みは細粒度に集約) の具体実装にあたる。

実装は mitmproxy + 自作 Python アドオンモジュール + 宣言的なポリシースキーマ。具体的に何ができるかは次節の表を参照。

## 2. 機能スコープ

| できること | 概要 |
|---|---|
| TLS 終端 | mitmproxy が CONNECT を受け、偽証明書を発行して https 通信の内容を監査 |
| CA 注入 (bootstrap) | 作業コンテナ起動時に `mitm.it/cert/pem` を取得し、OS トラストストア + 各言語ランタイムの環境変数 (Node / Python / pip / cargo / git) に統合 |
| ホスト × HTTP メソッド × パスの ACL | policy.json で宣言的に書く。原則拒否をベースに、3 段の区分 (`readonly_hosts` / `trusted_hosts` / `allow_rules`) のいずれかで明示的に許可 (詳細は §3.2) |
| ヘッダ注入 | プライベートレジストリの Bearer など、ホストを限定して `Authorization` をプロキシで足す |
| 監査ログ | 拒否 / ヘッダ注入を 1 行で標準エラーに出す。秘匿値そのものは出さない |
| プラグイン方式による拡張 | 本体は薄く保ったまま、利用側で固有の処理をアドオンとして追加できる |

## 3. 設計上の重要決定

### 3.1 読み取りのみ許可の表現と運用方針

3 段の区分 (trusted_hosts / readonly_hosts / allow_rules) のどれでも許可を書けるが、本リポジトリは **`readonly_hosts` を主として使う運用** を推奨する。これは [02-design.md](./02-design.md) §3.1 の運用原則 (粗粒度の mitm-proxy は読み取りのみを主とし、書き込みは細粒度に集約) の具体化にあたる。

この運用方針を取る理由:

- 参照系の HTTP メソッドは「副作用なし」のセマンティクスを持つため、ホストを信頼できれば一律で許可が成立する
- 更新系を mitm 層で個別に許可しようとすると、ポリシーがツール固有の内部実装 (裏側でどういう API を呼び出しているか) に依存する部分が大きくなり統制が難しくなりがち
- 「外部への更新」は MCP のような **操作粒度で語れる軸** に集約する方が、ACL 表現が簡潔かつ明瞭

例外的に MCP を介さない更新処理 (例: git push) を許可したいケースでは、用途特化のカスタムプロキシ (`recipes/git-gateway/`、[02-design.md](./02-design.md) §3.2) に切り出す選択肢もある。そうすることで、本プロキシのポリシーは `readonly_hosts` を中心に書ける形を保ちやすくなる。

### 3.2 policy.json の 3 区分

policy.json では以下の 3 区分で許可を書く:

| 区分 | 許可範囲 | 用法 |
|---|---|---|
| `readonly_hosts` | ホストを 1 つ書けば参照系 (GET / HEAD / OPTIONS) のみ通す | 大半の API / レジストリの読み取り用 (例: `api.github.com`, `registry.npmjs.org`) |
| `allow_rules` | (ホスト, パス, HTTP メソッド) のマッチング条件リスト | readonly ホストへの副作用のない POST など、特定の書き込みを限定的に許可 (例: pnpm audit の POST) |
| `trusted_hosts` | ホストを 1 つ書けば全 HTTP メソッド素通し | 全 HTTP メソッドの許可が要るエンドポイントだけに絞る (例: `api.anthropic.com`) |

3 区分のいずれにもマッチしないホストは原則拒否で落ちる。

### 3.3 利用側の拡張点: MITM_EXTRA_ADDONS

利用側で **追加アドオンを 1 ファイル単位でプラグインできる** 拡張点として `MITM_EXTRA_ADDONS` 環境変数を持つ。これにより:

- 本体は薄く保ったまま
- 利用側で固有の処理 (例: GitHub の git smart-HTTP に対する PAT 注入 + パスベースの push ACL) をアドオンとして追加できる

実例として `alternatives/git-mitm-proxy-addon/` は本イメージを継承し、`MITM_EXTRA_ADDONS=github` で `GitHubPolicy` を載せて、git-gateway を立てずに mitm 1 サービスで済ませる軽量代替を構成している (詳細は付録 [alt-git-mitm-proxy-addon](./appendix/alt-git-mitm-proxy-addon.md))。

「基本コンポーネントを薄く保ち、利用側で責務を足す」というプラグイン方式で、基本コンポーネントの責務肥大を抑えつつ利用側の柔軟性も担保する。

## 4. 「信頼境界」としての位置付け

[02-design.md](./02-design.md) §4 で並べたレシピ評価軸との対応関係を整理する:

| 評価軸 | mitm-proxy がどう満たすか |
|---|---|
| 秘匿情報は作業コンテナ外に置く | header_inject で注入する Bearer 等の秘匿情報はホスト側の env ファイル (`~/.config/devsbx/mitm-proxy.env`) に置き、`env_file` でプロキシコンテナの環境変数に渡す。作業コンテナ側のファイルシステムにも環境変数にも入らない。CA 秘密鍵も mitmproxy コンテナの名前付きボリュームに閉じ込め、作業コンテナには公開鍵側 (cert) のみ配布 |
| 作業コンテナはプロキシのみと通信する | 作業コンテナは internal ネットワークに閉じ、`HTTPS_PROXY` 経由でしか外向き通信できない ([03-foundation.md](./03-foundation.md) の二層ネットワークパターン) |
| ACL はプロキシ側で評価する | ホスト × HTTP メソッド × パスの ACL はプロキシ側のアドオンで評価。作業コンテナから書き換える経路は塞ぐ (詳細は §5「アドオン / ポリシー配置の注意」) |
| 境界ドメインは信頼できる先に限定する | trusted / readonly / allow_rules の 3 段で明示的に列挙されたホストだけが通る。原則拒否 |

mitm-proxy は **粗粒度・暗黙的な操作許可** を担う基本コンポーネントとして、4 つの評価軸すべてを満たす。mcp-proxy (細粒度・明示的、MCP ツール名で絞る) と直交する境界で、レシピごとに両者を組み合わせて使う ([02-design.md](./02-design.md) §3)。

## 5. アドオン / ポリシー配置の注意

なお、本イメージを継承して独自アドオンを載せる場合、**アドオンと policy.json は `COPY` でイメージレイヤに焼き込む** か、バインドマウントするなら **作業コンテナのマウント範囲外のパス** (例: `~/.config/.../my-policy.json`) に置くこと。

特に mitmproxy のアドオンは **hot reload する** ため、書き換え可能なパスに置くと変更が即座に取り込まれる。作業コンテナから書き換え可能な場所に配置すると、作業コンテナ内部から監視コードを書き換えられる経路が生まれ、ACL を変更する抜け穴になってしまう。

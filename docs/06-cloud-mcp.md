# レシピ: クラウド認証情報の短寿命化

ここからレシピ各論に入る。本章はクラウドプロバイダの MCP 連携を扱う `recipes/cloud-mcp-with-short-lived-credential/` を取り上げる。

## 1. 章のスコープ

クラウド認証情報は権限範囲が広く課金にも直結するため、他のレシピより扱いが重い。本章ではプロキシ内の認証情報自体を短寿命化する追加レイヤを入れる。具体例として Google Cloud MCP (gcloud-mcp) を `mcp-proxy` 経由で公開する形を取る。

## 2. クラウド認証情報固有のリスク

クラウド認証情報を重く扱う理由:

- **従量課金** — 呼び出しがリソース消費 / 課金に直結する
- **影響範囲が外部に表れる** — デプロイ / 公開バケット / IAM 変更などは外部から観測可能
- **長寿命認証情報の権限範囲** — 個人アカウントの ADC や SA 鍵は「アカウント全体」相当の権限を持ち、1 件の漏出で影響範囲が広い

一方でクラウド側はきめ細やかな権限管理と短寿命トークンの仕組みを持っているので、正しく使えば devcontainer 内に長寿命の認証情報を置かない設計が達成しやすい。

## 3. このレシピが追加する 2 つの方針

[02-design.md](./02-design.md) §1 の「操作許可をプロキシ群に集約する」に加えて、クラウド固有の方針を 2 つ重ねる。

### 3.1 個人の認証情報を devcontainer に渡さない

個人アカウントの認証情報をプロキシにも作業コンテナにも渡さず、目的別に権限を絞った専用の ID を介して必要な操作だけを許可する。GCP の場合、最小権限のサービスアカウントを作成し、個人アカウントから impersonation で短寿命トークンを発行する。devcontainer に流れる権限はこのサービスアカウントの IAM スコープに限定される。

### 3.2 プロキシにも長寿命の認証情報を置かない

他のレシピは「秘匿情報を作業コンテナ外に置く」までだが、クラウドではプロキシにすら **長寿命の認証情報を置かない**。プロキシが侵害されても、手に入るのは短寿命のアクセストークン (最大 1 時間) だけになる。

- ホスト側で発行された短寿命のアクセストークンを、ファイルとしてプロキシに渡す
- プロキシはトークンファイルを読み取り専用でマウントするだけ。リフレッシュトークンも SA 鍵も持たない
- トークンの更新はホスト側で行う (新しいファイルが上書きされる)

## 4. 構成

§3 の方針を GCP で実装した構成:

![クラウド認証情報の短寿命化の構成図](./cloud-mcp.excalidraw.png)

**プロキシ内にリフレッシュトークンは無い**。リフレッシュはホスト側 `refresh-token.sh` が担い、プロキシはトークンファイルを読むだけ。

## 5. 限界

- **寿命中のプロキシ侵害** — 寿命残分 (最大 1 時間弱) のサービスアカウントスコープが漏れる。影響範囲は専用サービスアカウントの IAM に閉じる
- **課金は止まらない** — IAM の許可範囲内なら重課金 API (BigQuery 等) も通る。budget alert / quota 制御の併用が要る

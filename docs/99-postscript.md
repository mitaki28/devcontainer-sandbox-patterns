# あとがき

本リポジトリで扱っている技術領域 — devcontainer のネットワーク隔離、プロキシ経由の外向き通信制御、独自 CA を発行して TLS 終端し通信内容を検査して ACL を効かせる構造 — は、見渡してみると、もともと **大企業の情報システム部門 (情シス) が組織内の従業員端末に対して敷いてきた制御** の系譜にある。エンタープライズ用語で言えば SWG (Secure Web Gateway) / SSL inspection / forward proxy / egress filter のような領域で、事例を検索しても、まずこの大企業向けの文脈の知見が目に入ってくる。それと同じ構造を、本リポジトリでは個人開発者の手元の devcontainer 1 つに対して組み直している。

なぜ個人レイヤーで同じ制御が要るようになったかと言えば、AI エージェント (Claude Code / Cursor / Codex CLI 等) が **任意コード実行能力を持って開発者の手元で動く** ようになったからである。組織内端末に対して情シスが想定してきた脅威 — 秘匿情報の外部持ち出し、認証情報の意図しない送信、許可外の更新操作 — は、AI エージェントと秘匿情報 + 任意の外向き通信が同居する devcontainer においてほぼ同型で再演される ([01-problem.md](./01-problem.md) §3 で見た構造)。AI エージェントの普及によって、情シス的な制御が扱ってきた問題領域が、組織レイヤーから個人開発者レイヤーまで降りてきた、とも言える。

需要側だけでは構成は組めない。実際に組み立てられる手段の側も同時に成熟していた、というのがもう一つの面白さだと思う。数十年前のエンタープライズ向けネットワーク制御は、商用プロキシ装置 / 商用 SSL inspector / 商用ファイアウォール + 専用 ACL 装置を組み合わせ、ハードウェア込みで数百万〜数千万のオーダーで調達し、専用ラックで構築するのが前提で、個人開発者が手元で試す対象ではなかった。それが本リポジトリで使っている材料を並べてみると、mitmproxy・Caddy・Squid・Docker / Docker Compose / Docker ネットワーク (`internal: true`)・Linux network namespace のような OSS で、個人の手元の PC で組み立てられる。これらは個別には特殊なツールですらなくなっていて、開発者の標準的なツールチェーンの近くに降りてきている。

この需要と供給のどちらか一方でも欠けていれば、本リポジトリのような構成は個人の手元で組み立てて試せる対象にならなかった。両方が同時に揃って初めて、個人開発者の試行として成立する。**面白い時代になったな** というのが、構築の過程で感じた率直な感想である。

# 動機: 踏み込んだ隔離要件を明確に記述する

AI エージェント (Claude Code / Cursor / Codex CLI) を日常的に使う開発者にとって、devcontainer は「エージェントの影響範囲をホストから切り離す手段」として推奨されている。本章では、devcontainer をベースに **どこまで踏み込んだ隔離要件を立てうるか** を整理し、それを 1 つの作業コンテナ内で実装しようとしたときに条件が分散する構造から、次章 ([02-design.md](./02-design.md)) で採る設計の動機を示す。

## 1. AI エージェント × devcontainer の前提

AI コーディングエージェントは **任意コード実行能力を持つ**。
ファイル編集だけでなく、ターミナルでコマンドを実行し、ネットワークに出て、外部 API を叩く。
これはエージェントの便利さの源泉だが、同時に **エージェントが動作する環境 = 任意コード実行されうる環境** であることを意味する。

devcontainer の役目は、その実行コンテキストを **ホスト OS と切り離すこと** にある。ホスト上の `~/.ssh/` や `~/.aws/` を直接読まれるリスクは、コンテナに閉じ込めれば一段下がる。

ただし「開発できる状態」を整える過程で、コンテナからの外向き通信は任意の宛先に対して開放され、その上に各種の秘匿情報・認証情報が流れ込んでくる。これにより、以下の 3 因子が同時に揃う:

- **任意コード実行能力** — エージェントの便利さの源泉として持つ (前提)
- **任意の外向き通信** — 依存取得 / 外部 API 呼び出しを成立させるため
- **長寿命の秘匿情報** — 外部サービスを操作するための認証情報 (具体例は §3.1)

この 3 因子の同居構造に対して、業界の既存対策がどこまで届くか (§2)、そして本書のレシピがそこからどう踏み込むか (§3) を、以下で順に見る。

## 2. 既存対策の適用境界

業界で標準的に提案される対策は以下:

- **一般的な AI エージェント向け devcontainer 構成例** (単一 Dockerfile + 任意で iptables / SNI ベース) — 作業コンテナ全体に境界を 1 つ引く。境界の単位はホスト粒度までで (CDN 配下の動的 IP は対象外、HTTP メソッド × パス粒度は表現できない)、秘匿情報は `mount` / `devcontainer.env` 等で渡す前提で作業コンテナ内に置く構造
- **個別の秘匿情報のスコープを絞る** (PAT スコープ / OAuth スコープ最小化等) — 漏洩後の被害範囲を限定する設計。本書のレシピでも必要に応じて併用する (例: `cloud-mcp-with-short-lived-credential` での短寿命認証情報発行 + 権限スコープ最小化)
- **ドメインレベル ACL** (iptables / SNI ベース) — 許可ドメイン外への通信は遮断できるが、許可ドメイン内の操作粒度は識別できない

これら標準対策は、それぞれの境界の中では効く設計である。3 因子のうち外向き通信をドメイン粒度で絞り、漏洩後の被害範囲を限定する用途にはここで足りる。

## 3. 踏み込んだ隔離要件の輪郭

本書のレシピは、標準的な構成 (§2) に加えて、(a) 秘匿情報自体の漏洩を防ぐ、(b) 許可ドメイン内の意図しない操作を困難にする、ところまで目指すとき、何が必要でどのような構成になるかを追求する技術探索である。

### 3.1 秘匿情報自体の漏洩を防ぐ

エージェントが外部システムを操作するための認証情報は、標準的には作業コンテナ内のホームディレクトリに保存される:

- **MCP トークン** — GitHub MCP の PAT、Atlassian MCP の OAuth リフレッシュトークン、Cloud MCP (GCP / AWS) の SA 鍵 / IAM 認証、各種 API キー型 MCP の固定トークン (`~/.config/<tool>/`, `~/.claude.json` 等)。MCP サーバ自体も作業コンテナ内で起動することが多く、エージェントと認証情報が同じプロセス境界に置かれる
- **Git 認証情報** — SSH 鍵 (`~/.ssh/`)、GitHub PAT、GPG 鍵 (`~/.gitconfig` の credential helper / `~/.config/git/credentials`)。SSH agent forward を使う場合でも、その経路自体は作業コンテナ内のプロセスから利用可能で隔離にはならない
- **パッケージレジストリ認証** — `.npmrc` / `.pypirc` / `~/.docker/config.json` / `~/.cargo/credentials.toml` 等

エージェントが任意コード実行能力を持つ以上、これらのファイルは事実上読み出せる。加えて、1 個あたりの権限範囲が広い (GitHub PAT 1 個でリポジトリ / 組織全権相当) ため、漏洩時の影響範囲も大きい。

ここで立てる要件は、**PAT / OAuth リフレッシュトークン等の秘匿情報が作業コンテナの中に存在しない** 状態。

### 3.2 許可ドメイン内の意図しない操作を困難にする

多機能なサービスのドメインを許可した瞬間、そのドメイン上の全 API (= 参照系と書き込み系の両方) が許可される構造になる:

- **`api.github.com`** — issue / PR の参照と同時に、コメント書き込み、Actions trigger、Webhook 設定、任意リポジトリへの push までが同じ経路で通る
- **`registry.npmjs.org`** — `npm install` のための参照と同時に、自身が公開しているパッケージへの `publish` / `deprecate` / `unpublish` までが同じ経路で通る

どの API が実際に呼び出されるかは、経路ではなくその経路を通る認証情報 (§3.1) やエージェントへの指示によって決まる。ドメイン粒度の許可は、ここでの選別を担えない。

ここで立てる要件は、**ドメインを許可しても、許可したい操作 (例: 指定したリポジトリへの push) だけが通り、それ以外の意図しない操作 (Actions trigger / 任意リポジトリへの push 等) は通らない** 状態。

## 4. 1 コンテナ内で実装しようとすると条件が複数のレイヤに分散する

§2 で見た個別対策を組み合わせて、§3 の踏み込んだ要件を 1 つの作業コンテナ内で漏れなく実装する道筋もある。ただし両軸とも、何を信頼すれば全体が成立するかが複数のレイヤに分散する。

秘匿情報軸 (§3.1) では、エージェントから秘匿情報への到達経路を塞ぐためにファイル ACL / プロセス権限 / カーネル層 (namespace 等) を同時に整える必要がある。

通信粒度軸 (§3.2) では、ホスト粒度を超えた許可を書くために HTTP プロキシでの L7 評価 / プロキシ設定の改ざん防止 / 通信元アプリの proxy 環境変数尊重を同時に整える必要がある。

どちらの軸でも、これらの条件が同時に正しく保たれていないと全体が崩れる。

## 5. 信頼境界の集約というアイデア

§4 で見た「複数面への分散」という困難に対し、本書は別方向のアプローチを採る: **信頼境界** (= 何が崩れたら全体が崩れるかの線) **を独立した Docker コンテナ群 (= プロキシ群) に集約する** 形に組み替える。これにより、分散していた条件が限定された面に揃い、見通しやすくなる。

具体的な設計と、その前提・保証は次章 [02-design.md](./02-design.md) で扱う。

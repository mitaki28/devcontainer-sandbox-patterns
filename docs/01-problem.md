# 問題: なぜ既存の devcontainer 隔離では足りないか

AI エージェント (Claude Code / Cursor / Codex CLI) を日常的に使う開発者にとって、devcontainer は「エージェントの影響範囲をホストから切り離す手段」として推奨されている。本章では、実際にそれを実用的な開発環境として整えていくと **どこで隔離が崩れるか** を整理し、次章で導入する設計原則 ([02-design.md](./02-design.md)) の必要性を示す。

## 1. AI エージェント × devcontainer の前提

AI コーディングエージェントは **任意コード実行能力を持つ**。
ファイル編集だけでなく、ターミナルでコマンドを実行し、ネットワークに出て、外部 API を叩く。
これはエージェントの便利さの源泉だが、同時に **エージェントが居る環境 = 任意コード実行されうる環境** であることを意味する。

devcontainer の役目は、その実行コンテキストを **ホスト OS と切り離すこと** にある。ホスト上の `~/.ssh/` や `~/.aws/` を直接読まれるリスクは、コンテナに閉じ込めれば一段下がる。

ただし「開発できる状態」を整える過程で、コンテナからの外向き通信は任意の宛先に対して開放され、その上に各種の秘匿情報・認証情報が流れ込んでくる。本章で見るのは、**この組み合わせが標準的な devcontainer 構成では止まらない** という点である。

## 2. 標準構成で残る 2 つの穴

実際に AI エージェント + devcontainer で日常開発をしようとすると、以下 2 つの構造が標準で組み上がる。

### 2.1 通信先制御がドメイン粒度までで止まる

devcontainer からの外向き通信はデフォルトで任意ホスト・任意ポートに出られる。これは `npm install` / `pip install` / `cargo build` / `docker pull` / `git fetch` / 任意の HTTP API 呼び出し すべてを成立させるためで、開発体験上自然な設定である。

任意の外向き通信を絞ろうとしても、標準的な手法 (iptables / SNI ベースの許可リスト) は **ドメイン粒度までしか効かない**。多機能なサービスの場合、これは「経路を 1 つ許可すると、そのサイトが提供する全 API への経路を同時に許可する」ことを意味する:

- **api.github.com を許可** → issue や PR の参照のために許可したつもりでも、同じドメインに issue / PR コメント、Actions trigger、Webhook 設定など書き込み操作が乗っている
- **registry.npmjs.org を許可** → `npm install` のために許可したつもりでも、同じドメインに `publish` / `deprecate` / `unpublish` が乗っている

どの API が実際に呼び出されるかは、経路ではなく **その経路を通る認証情報 (§2.2)** や **エージェントへの指示** によって決まる。ドメイン粒度の許可リストは、ここでの選別を担えない。

### 2.2 長寿命の秘匿情報がコンテナ内に居る

エージェントが外部システムを操作するための認証情報は、標準的にはコンテナ内のホームディレクトリに保存される:

- **MCP トークン** — GitHub MCP の PAT、Atlassian MCP の OAuth リフレッシュトークン (DCR)、Cloud MCP (GCP / AWS) の SA 鍵 / IAM 認証、各種 API キー型 MCP の固定トークン (`~/.config/<tool>/`, `~/.claude.json` 等)。MCP サーバ自体もコンテナ内で起動することが多く、エージェントと認証情報が同じプロセス境界に置かれる
- **Git 認証情報** — SSH 鍵 (`~/.ssh/`)、GitHub PAT、GPG 鍵 (`~/.gitconfig` の credential helper, `~/.config/git/credentials`)。SSH agent forward を使う場合でも、その経路自体は AI エージェントのプロセスから利用可能で隔離にはならない
- **パッケージレジストリ認証** — `.npmrc` / `.pypirc` / `~/.docker/config.json` / `~/.cargo/credentials.toml` 等

エージェントが任意コード実行能力を持つ以上、これらのファイルは事実上エージェントから読み出せる。読み出せた瞬間、§2.1 の任意の外向き通信経路で外部に送出できる。

加えて、これらの秘匿情報は **1 個あたりの権限範囲が広い** ことが多い (GitHub PAT 1 個でリポジトリ / 組織全権相当) ため、漏洩時の影響範囲が大きい。

## 3. 既存対策の限界

これらの穴を埋めるために、各所で部分的なアプローチが提示されている。が、どれも「コンテナ内で頑張る」延長線上にあり、構造的に限界がある。

### 3.1 公式 devcontainer サンプル: Dockerfile + iptables

公式の devcontainer サンプルは単一 Dockerfile + 任意で iptables / ファイアウォールという構成。

- iptables / ファイアウォールは **ドメイン名 ACL に弱い** (CDN 配下の動的 IP は追えない)
- 秘匿情報は `mount` / `devcontainer.env` 等で渡す前提で、コンテナ内に置く構造を肯定する
- 各認証経路 (git / MCP / レジストリ) の隔離はスコープ外

これは「コンテナ全体に強い境界を 1 つ引く」発想で、その境界の **内側に秘匿情報が居る限りエージェントから読める** という前提を疑わない。

### 3.2 個別の秘匿情報のスコープを絞る

「API トークンのスコープを絞る」「OAuth スコープを最小化する」等の対策は **被害範囲を絞るが、漏洩自体は防がない**。
エージェントがトークンを読めて任意送信できる以上、トークン値そのものが攻撃者の手元に渡る。
スコープ絞りは事後の被害低減であり、本章で問題にしている「コンテナ内に秘匿情報 + 任意の外向き通信が同居する構造」を解消しない。

## 4. 構造的な観察

§2 で見た 2 つの穴に共通するのは、次の 3 因子が同時に揃う構造である:

- **コンテナから任意の宛先への通信が出ている** (許可リストが実用解像度で書けない)
- **長寿命の秘匿情報がコンテナ内にある** (API トークン / OAuth リフレッシュトークン / SSH 鍵 / レジストリ認証)
- **エージェントが任意コード実行能力を持つ** (これは前提)

この 3 つが揃った時、漏洩は構造的に 1 ステップで成立する。
「コンテナ内に良い iptables を書く」「秘匿情報のスコープを絞る」はいずれもこの 3 つのうちのどれかを **量的に弱める** だけで、**質的には残す**。

エージェントの任意コード実行能力を取り下げることはできない以上、残る選択肢は:

- **任意の外向き通信をプロキシ経由に強制する** (許可リストをプロキシ側で評価する)
- **長寿命の秘匿情報をコンテナの外に出す**

の 2 つで、これがそのまま次章 [02-design.md](./02-design.md) で扱う「信頼境界をプロキシ群に置き、作業コンテナ自身は認可を制御できない」設計の出発点となる。

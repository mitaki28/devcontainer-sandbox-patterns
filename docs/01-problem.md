# 動機: 踏み込んだ隔離を見通しよく実現したい

AI エージェント (Claude Code / Cursor / Codex CLI) を日常的に使う開発者にとって、devcontainer は「エージェントの影響範囲をホストから切り離す手段」として推奨されている。本章では、もう一歩踏み込んだ隔離として何を目指すかを整理し、それを見通しよく実現するために次章 ([02-design.md](./02-design.md)) で採る設計の動機を示す。

## 1. AI エージェント × devcontainer の前提

AI コーディングエージェントは任意コード実行能力を持つ。
ファイル編集だけでなく、ターミナルでコマンドを実行し、ネットワークに出て、外部 API を叩く。
これはエージェントの便利さの源泉だが、同時に、エージェントが動作する環境 = 任意コード実行されうる環境であることを意味する。

devcontainer の役目は、その実行コンテキストをホスト OS と切り離すことにある。ホスト上の `~/.ssh/` や `~/.aws/` を直接読まれるリスクは、コンテナに閉じ込めれば一段下がる。

ただし「開発できる状態」を整える過程で、コンテナからの外向き通信は任意の宛先に対して開放され、その上に各種の秘匿情報・認証情報が流れ込んでくる。

この前提のもとで、標準的な対策でどこまで守れるか (§2)、そして本書のレシピがそこから何を目指すか (§3) を、以下で順に見る。

## 2. 標準的な対策の守備範囲

AI エージェント向け devcontainer の典型的な構成 (単一 Dockerfile に作業コンテナを定義し、必要に応じて外向き通信制御を加える形) では、次の対策が標準的に取られる:

- **個別の秘匿情報のスコープを絞る** (PAT スコープ / OAuth スコープ最小化等) — 漏洩後の被害範囲を限定する設計
- **ドメインレベル ACL** (iptables / SNI ベース) — 許可ドメイン外への通信は遮断できるが、許可ドメイン内の操作粒度は識別できない

## 3. 本書の目標

本書のレシピは、標準的な構成 (§2) からさらに踏み込んで、秘匿情報自体を作業コンテナの外に出す / 許可ドメイン内でも操作を絞る、ところまで目指す。

### 3.1 秘匿情報自体を作業コンテナの外に出す

エージェントが外部システムを操作するための認証情報は、標準的には作業コンテナ内に保存される:

- **MCP トークン** — GitHub PAT、Atlassian OAuth トークン、GCP の SA 鍵等
- **Git 認証情報** — SSH 鍵、PAT、credential helper 等
- **パッケージレジストリ認証** — `.npmrc` / `.pypirc` 等

エージェントが任意コード実行能力を持つ以上、これらのファイルは読み出される可能性がある。そして長寿命な秘匿情報は、一度漏洩すると、更新するまで使われ続けてしまう。

こうした状況に対し、本書のレシピは **秘匿情報自体を作業コンテナの外に出す** ことを目指す。

### 3.2 許可ドメイン内でも操作を絞る

多機能なサービスのドメインを許可すると、参照系だけでなく書き込み系の通信経路もネットワーク上は開く:

- **`api.github.com`** — issue の参照のために許可すると、コメント書き込みや任意リポジトリへの push の経路も開く
- **`registry.npmjs.org`** — `npm install` のために許可すると、`npm publish` の経路も開く

実際に操作が成立するかは認証情報 (§3.1) のスコープ次第だが、ドメインを許可した時点でネットワーク層の防御は効かなくなる。

こうした状況に対し、本書のレシピは **許可ドメイン内でも操作を絞る** ことを目指す。例えば、指定したリポジトリへの push は通すが、任意リポジトリへの push は通らない形にする。

## 4. プロキシ群への集約

§3 の目標は作業コンテナ内でも追求できるが、任意コード実行が前提の環境内で秘匿情報や許可判定を守るには、ファイル権限・実行ユーザー分離・プロセス隔離等を組み合わせて到達経路を塞ぐ必要があり、考慮すべきことが多い。秘匿情報と許可判定を別の Docker コンテナに分離すれば、コンテナ境界で到達経路が断たれるため、より単純な構造で同じ目標を実現できる。

本書では、この分離を独立した Docker コンテナ群 (プロキシ群) として構成する。具体的な設計は次章 [02-design.md](./02-design.md) で扱う。

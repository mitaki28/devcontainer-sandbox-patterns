# レシピ: 開発サーバをホストブラウザに見せる

ここまでは作業コンテナから外への外向き通信をプロキシに集約するレシピだった。本章はそれと **直交するインバウンド軸** — 作業コンテナ内で動く開発サーバ (`pnpm dev` の `:3000` 等) をホストブラウザから見るための経路 — を扱う。

## 1. 章のスコープ

これまでのアウトバウンド軸レシピ (cloud / web-fetch / git) は作業コンテナから外への通信をプロキシに集約する構造だった。実用上、作業コンテナ内の開発サーバ (Next.js / Vite / Django / `pnpm dev` の `:3000` 等) をホストブラウザから見たい場面が頻繁にあり、本章はこの **インバウンド (外 → 内) 経路** を扱う。

本章で扱うのは「経路の作り方の選択肢の整理」と「インバウンド軸でのレシピ評価軸の読み替え」のみで、個別実装の詳細は各レシピ / 付録に譲る。

## 2. なぜリバースプロキシ層が要るか

[03-foundation.md](./03-foundation.md) で示した通り、作業コンテナは `internal: true` なネットワークに閉じ込めている。`internal: true` は外向きの通信だけでなく **ホスト側からのポート公開も無効化する** ため、`internal: true` 単独所属のコンテナに `ports:` を書いても公開は効かない。インバウンド経路を作るには、アウトバウンド側にプロキシ群を置いたのと同様、**インバウンド側にも別コンテナのリバースプロキシ (Caddy) を立て、そこだけがホストへのポート公開を担う** 形に分離する。ホストブラウザは `127.0.0.1:8080` 経由で `Host` ヘッダによって作業コンテナの任意のポートにルーティングされる。

実装は単独起動と並列起動の 2 つに分かれ、後者の動的ルーティング (Docker DNS を活用して Caddy 設定を静的に保つ構造) は [11-multi-workspace.md](./11-multi-workspace.md) で扱う。

- [`recipes/ingress-single-workspace/`](../recipes/ingress-single-workspace/) — 単独起動の最小構成
- [`recipes/ingress-multi-workspace/`](../recipes/ingress-multi-workspace/) — 並列起動向けに動的ルーティングを持つ構成 (詳細は [11-multi-workspace.md](./11-multi-workspace.md))

> [!NOTE]
> なお IDE 経由でしか開発サーバを叩かない用途では、別コンテナを立てずに **IDE 内蔵 port forward** で済ませる選択肢もある。
> 
> 具体的には、 `devcontainer.json` で `forwardPorts: [3000, 4000]` などを記述し、開発サーバを直接 forward すれば、それだけホスト向けにポートを公開できる。（ポートの衝突についても IDE 側で自動解決されるが、公開されるポート番号は動的に変わりうる点には注意。）
> 
> しかし、本書においては、より汎用性の高い構成例として別コンテナでリバースプロキシを立ち上げる方針を採用している。これは以下の理由によるもの:
> 
> - **構成の対称性** - 本書はアウトバウンド軸を別コンテナのプロキシ群として組み上げてきたため、インバウンド軸も同じく別コンテナのプロキシとして並列に示したい
> - **devcontainer 以外の環境への応用性** - 本リポジトリの `compose.yaml` は devcontainer を介さずとも実際には単体で運用可能な形となっており、インバウンド軸についても IDE 非依存とした方が運用の幅が広がる
> - **特定の devcontainer 実装に対する非依存性** - `forwardPorts` は devcontainer 実装によってサポート状況がまちまちであり、例えば devcontainer CLI (`@devcontainers/cli`) ではサポートされない
> 
> IDE 内蔵 port forward で済ませる場合、次章以降の本書のインバウンド軸に関する議論は読み飛ばしても問題ない

## 3. インバウンド軸の位置付け

[02-design.md](./02-design.md) §2 の脅威モデル (作業コンテナの侵害から秘匿情報持ち出し / 任意の権限拡張を防ぐ) は基本的にアウトバウンド軸を想定したものであり、インバウンド軸は脅威対策ではなく **ホストブラウザから作業コンテナの開発サーバへ到達する経路を作るための章** として位置付ける。

その上で、本軸でも維持される構造的な性質を 2 つ挙げておく:

- **作業コンテナは自分のポートをホストに公開できない** — `internal: true` 単独所属の効果
- **リバースプロキシは LAN から隔離されている** — ホストポートは `127.0.0.1:8080` にバインドし、LAN 上の他デバイスからは到達できない

これらは「信頼境界をホスト側に置く」本書の構造から方向を変えても自然に出てくる性質となっている。

## 4. 限界: リバースプロキシ自身の侵害

リバースプロキシは internal に加えて、ホストへのポート公開のため external にも所属する。この構造のため、侵害されると外向き通信 + 作業コンテナへの到達が同時に成立する。緩和策はイメージ最小化 (`caddy:2-alpine` 等) とリバースプロキシに秘匿情報 / トークンを載せないことに限られる。

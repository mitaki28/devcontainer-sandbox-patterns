# 設計: 操作許可をプロキシ群に集約する

前章 ([01-problem.md](./01-problem.md)) では、秘匿情報と許可判定を作業コンテナの外に出す動機を示した。本章では、これをプロキシ群に集約する設計と、その前提を示す。

## 1. 基本設計

本リポジトリは、**操作許可をプロキシ群に集約する** 構造を取る。

ここで言う **操作** は、外部に影響を与える操作 (HTTP リクエスト / git コマンド / MCP 呼び出し) を指し、作業コンテナ内部のユーザー権限やファイル ACL 等は含まない。AI エージェントが root であっても、その操作が作業コンテナの内部に閉じている限り、作業コンテナ外から見れば影響はない。逆に非 root でも、GitHub の issue にコメントできる PAT を持っていれば、そのコメントは作業コンテナの外に直接届く。重要なのは、内部権限の有無ではなく **外部影響の有無** である。

**許可** は、「何を通すかの判定」と「その実行に必要な秘匿情報の保持」の両方を含む。両者は表裏一体で、秘匿情報を持っていれば操作が通るので、秘匿情報を作業コンテナに置くこと自体が操作許可を渡すことになる。そして、漏洩すれば作業コンテナ外から直接使われ、プロキシを迂回できてしまう。

これらを作業コンテナの外 (独立した Docker コンテナ群 = プロキシ群) に集約すると、次の構造になる:

- プロキシ群だけが秘匿情報を保持する (作業コンテナには入らない)
- プロキシ群だけが許可判定する (作業コンテナから書き換えられない)
- 作業コンテナは「操作」だけをプロキシ群に送る (秘匿情報は伝えない)
- 作業コンテナから外への通信は必ずプロキシ経由 (迂回路を持たない)

万一作業コンテナ内が乗っ取られても、外部への操作はプロキシ群が許可する範囲に閉じる構造になる。

## 2. 前提

本書の設計は Docker のネットワーク隔離 (`internal: true`) によって、作業コンテナがプロキシ群のみと通信可能な状況を作れることを前提としている。具体的には [03-foundation.md](./03-foundation.md) §5 で扱う。

もう 1 つの前提として、通信先サービス (github.com / registry.npmjs.org や各 MCP 実装など) を信頼できる対象として扱う。脆弱性がないことに加え、参照系メソッド (GET / HEAD / OPTIONS) が副作用を持たず、リクエスト内容が外部に露出しないことを含む。これが参照系をホスト単位で許可する方針 (§3.2) の土台になる。

本リポジトリのプロキシ実装 (mcp-proxy / mitm-proxy / git-gateway など) は、こういうプロキシがあると操作許可を整理しやすい、という考えのもとに学習も兼ねて最小構成で自作した実装例であり、実用品質やセキュリティは保証しない。§7 に挙げた既存実装に差し替えても同じ設計が成立する。

なお、以下は本書はあくまでも作業コンテナに対する操作許可として実現できる範囲までをスコープとしており、実運用上、設計目標を達成するためには多くの考慮事項が残る:

- **通信先サービスの信頼前提** — 実際にはアクセス解析等を通じてリクエスト内容が外部に露出したり、参照系メソッドで副作用を持つエンドポイント（webhook トリガーや GraphQL の一部実装など）が存在しうる。流量や影響範囲は限られることも多いが、§2 の前提が成り立たない場面はありうる
- **AI エージェント実行系 (Claude Code) 自身の認証情報** — 月額プラン経由では認証フローが非公開で、プロキシ介入が利用規約上グレーになりうるため、認証情報は作業コンテナ内に置く形になる。API トークン方式 (Bedrock / Vertex 等) なら原理的には適用できる
- **許可した操作の先で起きること** — 例えば push を許可した場合、push 先で CI/CD が走ればそこでの任意コード実行や情報漏洩はプロキシでは止められない。詳細は [08-git-gateway.md](./08-git-gateway.md) §6 を参照
- **外部から読める宛先への操作許可** — 公開リポジトリへの PR 作成のように、外部から読み取れる宛先への操作を許可すると、それ自体が情報を持ち出す経路になる

## 3. 操作許可の方針と基本コンポーネント

§1 では本書の構造を示した。本節では「外部への操作をどう許可するか」の方針と、それを担う基本コンポーネントを示す。

### 3.1 原則拒否 + 明示的な許可リスト

出発点は **原則拒否** である。作業コンテナは Docker `internal: true` ネットワークだけに所属し、外部への通信路を持たない ([03-foundation.md](./03-foundation.md))。外部に対する操作はすべて、プロキシ群を介して **明示的な許可リストを書く** ことでしか成立しない。本リポジトリでは、許可リストを書くための基本コンポーネントを 2 種類用意し、レシピごとに組み合わせる:

| 基本コンポーネント | 許可単位 | 粒度 | 操作許可の表現 | 主用途 |
|---|---|---|---|---|
| **mcp-proxy** | MCP ツール名 | 細 | 明示的 (「この操作を許す」を直接書く) | 認証 + 操作粒度を絞れるもの (GitHub, Atlassian, GCP, Context7 等) |
| **mitm-proxy** | ホスト × HTTP メソッド × パス | 粗 | 暗黙的 (通信が成立 = その通信が表す操作も成立) | 雑多な参照系 (`pnpm info`, レジストリ GET, ドキュメント参照等) |

### 3.2 粗粒度の操作許可は原則読み取りのみ許可

一般に HTTP 通信制御で操作を許可する方式は、ホストを 1 つ許可するだけで多くの操作 (読み取り / 書き込み / 副作用のあるものも含む) が通りうる。一方で、本リポジトリの mitm-proxy はホスト × HTTP メソッド × パスまで絞れるので、この粒度を活かして以下の運用方針を置く:

- **mitm-proxy は読み取りのみ許可を主とする** — ホスト許可は参照系 (GET / HEAD / OPTIONS) のみに絞ることを推奨する。読み取り操作だけに絞れる区分を優先することで、暗黙的許可の影響範囲を読み取り系に限定する
- **更新操作は mcp-proxy 経由で通す** — 外部への更新操作 (POST / PUT / PATCH / DELETE) は原則 mcp-proxy を通す。操作粒度で明示的な許可リストを書けるため、何が許されているかが追跡しやすい

読み取りのみ許可が成り立つ前提と運用の詳細は [05-mitm-proxy.md](./05-mitm-proxy.md) §3 を参照。

### 3.3 用途特化のカスタムプロキシ

mcp-proxy / mitm-proxy の粒度では表現しきれないケースもある。例えば git push の ref ACL は、MCP ツール名でもホスト × HTTP メソッド × パスでも書けない。こうしたケースには **用途特化のカスタムプロキシを別レイヤとして用意する** ことで対処できる。本リポジトリでは git transport 用に `git-gateway` を用意している。

詳細は [08-git-gateway.md](./08-git-gateway.md) を参照。

## 4. 別の視点: 作業コンテナに対する操作許可の枠組み

§1〜§3 で示した構造は、視点を変えれば **作業コンテナに対する操作許可の枠組み** として読める。各プロキシコンテナが「どの操作を通すか」と「そのための秘匿情報」をセットで持つので、コンテナの追加・削除がそのまま操作許可の追加・削除になる。例えば GitHub MCP プロキシを構成から外せば、GitHub への操作は丸ごと消える。

また、MCP / HTTP / git といった異なるプロトコルの操作許可が、それぞれ専用のプロキシコンテナとして併存できる。新しいプロトコルや通信先が必要になれば、対応するプロキシを足すだけで作業コンテナに与える操作許可を拡張できる。

本書のレシピは細かい粒度で絞る構成を示しているが、要件に応じてプロキシを減らしたり許可範囲を広げたりする構成も同じ枠組みで書ける。

## 5. 組み合わせ可能なアプローチ

本書のレシピと別軸で組み合わせ可能なアプローチ。

- **権限最小化** (PAT / OAuth スコープの最小化等) — 漏洩後の被害範囲を限定する
- **コンテナランタイム層の隔離** (rootless / seccomp / AppArmor / gVisor) — コンテナ侵害時の影響範囲を絞る
- **L3/L4 層での外向き通信制御** (Tailscale / Cloudflare WARP / WireGuard 等) — 外向き通信の出口を別ネットワークに集約する
- **VM 隔離** (Docker Sandbox / GitHub Codespaces / Firecracker 等) — より強い隔離を得る。なお、プロキシ群と作業コンテナの分離という構成はそのまま応用できる。

## 6. 適用範囲

本リポジトリは **個人開発者ローカルでの devcontainer 利用** を主眼とする。

- チーム共有環境 / CI / 本番環境への直接適用は想定外
- 本書のプロキシ群 (基本コンポーネント + 用途特化のカスタムプロキシ) が HTTP(S) ベースである前提のため、SSH / TCP-only / UDP は適用範囲外

SSH 経由の git や DB 直接接続が必要な場合、L3/L4 プロキシを別軸として追加する方法はあるが、本リポジトリでは扱わない。

# 設計: 信頼境界をプロキシ群に置き、作業コンテナ自身は認可を制御できない

前章 [01-problem.md](./01-problem.md) で、標準的な devcontainer 構成には「通信先制御がドメイン粒度までで止まる」「長寿命の秘匿情報がコンテナ内に同居する」という 2 つの構造的な穴があることを整理した。本章では、それらを統一的に塞ぐための設計原則と、本リポジトリのレシピがどの安全性モデルの下で作られているかを述べる。

## 1. 基本パターン

### 1.1 プロキシ群 = サービス、作業コンテナ自身は認可を制御できない

公式の devcontainer サンプル (単一 Dockerfile + iptables 程度) は、**作業コンテナ内部に秘匿情報を置き、内部で制御する** 発想を取る。AI エージェントが任意コード実行能力を持つ前提に立った上で、この方針で本書と同等の制御 (ドメインより細かい粒度の通信制限 + 秘匿情報を AI エージェントから到達不能にする) を組もうとすると、信頼境界の成立条件が **複数の独立した点に散らばる**。アプリケーション設定 / プロセス権限 / ファイル ACL / カーネル層の通信制御 / ユーザー権限設定 (sudo / setuid 等) — そのそれぞれが同時に正しく保たれる必要があり、どれか 1 つが崩れれば全体が崩れる。何を信頼すれば全体が成立するかを明確化しづらい構造になる。

本リポジトリは逆に、**信頼境界を独立した Docker コンテナ群 (= プロキシ群) に集約する** 形を取る。信頼境界が「プロキシ群の実装 + Docker ネットワーク設定」という限定された面に揃うため、何を信頼すれば全体が成立するかが明確化できる (具体的な内訳は §2):

- **プロキシ群だけが秘匿情報を読む** — API トークン / OAuth リフレッシュトークン / クラウドの認証情報はホスト側に置き、プロキシ群だけが読み込む。**作業コンテナのファイルシステムにも環境変数にも秘匿情報は入らない**
- **プロキシ群だけが ACL 判定する** (許可ホスト / 許可 ref / 許可 MCP ツール)
- **作業コンテナは「行動の意図」だけをプロキシ群に送る** (HTTP リクエスト / git コマンド / MCP 呼び出し)
- **作業コンテナから外への通信はすべてプロキシ経由で、それ以外の経路を Docker ネットワーク設定で塞ぐ** (`internal: true`)

### 1.2 本書における「認可」の定義

本書で「認可」と書くとき、これは **作業コンテナ外に影響を与える操作の許可** を指し、作業コンテナの内部で完結する操作は含まない。

たとえば、作業コンテナ内で AI エージェントが root 権限を持っていても、その操作が作業コンテナの内部に閉じている限り、作業コンテナ外から見れば影響はない。逆に、非 root ユーザーであっても、公開されているライブラリの main ブランチに push できる GitHub PAT を保持していれば、push 内容がそのままライブラリの新バージョンとして公開され、外部 (利用者環境) に直接影響を与えうる。

作業コンテナ (AI エージェント) に与える権限を語るうえで本質的なのは、内部権限の有無ではなく **外部影響の有無** である。次節で扱う「権限を渡せない / 権限を増やせない」の 2 軸も、この外部影響を軸に考える。

### 1.3 「作業コンテナ自身は認可を制御できない」の意味

「認可を制御できない」というのは、具体的に次の 2 軸を意味する:

- **権限を渡せない** — 作業コンテナの中に API トークン / OAuth リフレッシュトークン / SSH 鍵を一切置かない。自身に与えられた認可を外部に渡す経路が無い
- **権限を増やせない** — 作業コンテナから外への通信は internal ネットワークで Docker ネットワーク設定により遮断され、プロキシ以外に出られない。自身に与えられた認可範囲を拡張する経路が無い

この 2 軸が揃うことで、万一作業コンテナ内の AI エージェント / シェル / プロセスが乗っ取られても、攻撃者が外部に持ち出せるのはプロキシが許可する範囲に閉じる。

### 1.4 操作許可のモデル

§1.1〜1.3 では骨格 (信頼境界をプロキシ群に置く + 作業コンテナ自身は認可を制御できない) を述べた。次に「外部への操作をどう許可するか」のモデルを示す。

出発点は **原則拒否 (default deny)** である。作業コンテナを Docker `internal: true` ネットワークに閉じ込めることで、作業コンテナから外部への通信路自体を Docker ネットワーク設定で消す ([03-foundation.md](./03-foundation.md))。この時点で、作業コンテナから外部に対するあらゆる操作 (HTTP リクエスト / MCP 呼び出し / git push / `npm install` / 任意 API 呼び出し) は何一つ通らない。

したがって、外部に対するすべての操作はプロキシ群を介して **「明示的な許可リストを書く」** ことでしか成立しない。本リポジトリでは、許可リストを書くための **基本コンポーネント** を 2 種類用意し、レシピごとに組み合わせる:

- **mcp-proxy** — 細粒度・**明示的** な操作許可。MCP ツール名で「この操作を許す」を直接書く
- **mitm-proxy** — 粗粒度・**暗黙的** な操作許可。ホスト × HTTP メソッド × パスで通信を許可し、その通信が表す操作 (`pnpm info <pkg>` / API GET / etc.) が暗黙的に許可される

両者は粒度と許可表現の方向性が異なるため、レシピごとに **どこを細粒度で書き、どこを粗粒度で済ませるか** を組み合わせる。具体的な対比と運用原則は §3 で扱う。

この 2 つでカバーできない領域 (例: git push の ref ACL は MCP ツール名でもホスト × HTTP メソッド × パスでも捌けない) には、**用途特化のカスタムプロキシを自作する** という手も取れる。本リポジトリでは git transport 用に `git-gateway` を自作している (§3.2)。

## 2. 本書で扱う安全性モデル

本リポジトリは、§1 の原則拒否 + 明示的な許可リストという構造を採ることで、**どこまでが許可され、何に信頼を置いているかを明確化できる** ことを目指す。具体的には、次の 3 層に脆弱性がない限り、想定脅威モデル下で情報漏洩を起こさないように設計されている:

1. **基盤技術** — Docker engine / Docker Compose / Docker ネットワーク (`internal: true` による外部遮断)
2. **プロキシ / ゲートウェイ実装** — `lib/mcp-proxy` / `lib/mitm-proxy` / `git-gateway`
3. **境界の上流** — ドメイン提供者 (github.com / atlassian.com / registry.npmjs.org / accounts.google.com 等) と、mcp-proxy でラップする MCP 実装 (公式 GitHub MCP / Atlassian MCP / Context7 / gcloud-mcp 等)。これらは脆弱性が無いことに加え、**提供者自身を信頼できる** こと

3 層のどれかが崩れた時点でこの保証は崩れる。これは弱点ではなく、**「どこまでが本リポジトリの責任で、どこからが基盤・プロキシ実装・上流の責任か」を明示する境界線** として位置付けている。

条件 2 (プロキシ / ゲートウェイ実装) について本リポジトリが取るアプローチは、**自作領域を境界制御に必要な範囲まで絞り、外部依存を `@modelcontextprotocol/sdk` / mitmproxy / Caddy / `git-http-backend` のような標準コンポーネントに集約する** こと。プロキシ自身が信頼境界の中核を担う構造上、自前ロジックを増やすほど条件 2 の脆弱性面が増えるため、各プロキシはこの方針を共通の設計指針とする (具体的な集約構造は [04](./04-mcp-proxy.md) §5、[05](./05-mitm-proxy.md)、[08](./08-git-gateway.md) を参照)。

### 2.1 想定する脅威モデル

「想定する脅威モデル」とは具体的に以下を指す:

- 作業コンテナ内の AI エージェント / 拡張 / 依存パッケージが任意コード実行能力を持っている
- ホスト側ユーザー (= 開発者本人) は信頼できる
- ホスト側カーネル / Docker デーモンは信頼できる
- 攻撃者は外部ネットワーク経由・サプライチェーン経由で作業コンテナに到達する

この脅威モデルにおいて、本リポジトリのレシピは:

- 秘匿情報 (API トークン / OAuth リフレッシュトークン) の漏洩経路を絶つ
- 任意ホストへの外向き通信を絶つ
- 許可外操作 (任意リポジトリへの push / 任意 MCP ツール呼び出し) を絶つ

### 2.2 「上流を信頼」の意味について

条件 3 の「上流を信頼できる」は、プロキシ群 → 上流の通信において、次の経路で上流組織が保持する情報の扱いを上流の運用に委ねることを含む:

- 上流のウェブ解析やアクセスログに残る User-Agent / IP / 訪問パターン等のメタデータ
- 上流 API への引数として送信した内容 (検索クエリなど)

本リポジトリは、外部に流出する経路の遮断は責任範囲とするが、これら上流側に保持される情報の扱いは上流提供者の運用に委ねる。この前提を置きたくない用途には本リポジトリは適合しない。

### 2.3 このモデルを満たさない領域は対象外とする

たとえば「LLM に任意の URL を取得させたい」というユースケースは、上記条件 3 (境界の上流が限定される) が成立しない (= 任意ホストを許可した時点で境界が失われる) ため、**本リポジトリの設計範囲外** とする。任意ホスト fetch を扱うには、本書の安全性モデルとは別の保証の枠組み (Safe Browsing 等の事前チェック、LLM ベースの URL 判定など) が必要になる。

この境界線を明示することで、各レシピは「思想的にはこのモデルを守る」「未完成・別の枠組みが要る」を読者が判断できるようにする。本リポジトリでは前者だけを採用範囲とした (詳細は付録 [incomplete-fetch-mcp.md](./appendix/incomplete-fetch-mcp.md))。

## 3. 基本コンポーネント: mcp-proxy と mitm-proxy

§1.4 で導入した 2 種類の基本コンポーネントを、許可単位と粒度の観点で具体的に対比する。

| 基本コンポーネント | 許可単位 | 粒度 | 操作許可の表現 | 主用途 |
|---|---|---|---|---|
| **mcp-proxy** | MCP ツール名 | 細 | 明示的 (「この操作を許す」を直接書く) | 認証 + 操作粒度を絞れるもの (GitHub, Atlassian, GCP, Context7 等) |
| **mitm-proxy** | ホスト × HTTP メソッド × パス | 粗 | 暗黙的 (通信が成立 = その通信が表す操作も成立) | 雑多な参照系 (`pnpm info`, レジストリ GET, ドキュメント参照等) |

両者は **直交する境界** (mcp-proxy は「何を MCP に頼むか」、mitm-proxy は「どこに HTTP を投げるか」) で、包含関係にはない。レシピごとに、許可したい操作の性質に応じて両方を組み合わせる。

### 3.1 原則: 粗粒度の mitm-proxy は読み取りのみ許可を主とし、書き込みは細粒度に集約

mitm-proxy は通信制御で操作を暗黙的に許す方式なため、ホストを 1 つ許可するだけでそのホスト上のサービスに対するあらゆる操作 (読み取り / 書き込み / 副作用のあるものも含む) が通る。粒度が粗いほど許可範囲が広く取られ、制御も難しくなるリスクが存在する。

一般的な粗粒度の外向き通信制御 (Squid / iptables) はホスト単位までで止まるが、本リポジトリの mitm-proxy はホスト × HTTP メソッド × パスまで絞れる。これにより、許可を読み取り操作のみに絞る運用 (= ドメインを許可しても更新系操作は通さない) が選択肢として成立する。

この表現力を活かすため、本リポジトリは以下の運用原則を置く:

- **外部への更新操作 (POST / PUT / PATCH / DELETE) は原則 mcp-proxy 軸を通す** — 操作粒度で明示的な許可リストを書けるため、何が許されているかが追跡しやすい
- **mitm-proxy でホストを許可するときは、原則として読み取りのみ許可 (参照系 GET / HEAD / OPTIONS のみ) を主として推奨する** — mitm-proxy は原則拒否 + 明示的な許可リストで動作する。許可を書く際の選択肢のうち、読み取り操作だけに絞れる区分を優先することで、暗黙的許可の影響範囲を読み取り系に限定する
- **書き込みが要る場合は明示的に個別許可する** — 信頼ホストの全メソッド許可、特定パス × メソッドの個別許可 (例: pnpm audit の副作用のない POST)、アドオンによる独自ロジック追加 (例: [`alternatives/git-mitm-proxy-addon/`](../alternatives/git-mitm-proxy-addon/) で git push を通す)、などの形を取る

この原則があると、2 つの基本コンポーネントが同じホストを扱う場面 (GitHub: MCP + git push / GCP: gcloud-mcp + SDK 直叩き) で重複が交通整理される。

具体的なポリシー構造 (trusted_hosts / readonly_hosts / allow_rules の 3 区分) や個別許可の書き方は [05-mitm-proxy.md](./05-mitm-proxy.md) §3.2 を参照。

### 3.2 基本コンポーネントでカバーできない領域: 用途特化のカスタムプロキシ

mcp-proxy / mitm-proxy は **汎用** の基本コンポーネントで、対象バックエンドを差し替えるだけで多様なユースケースに適用できる (mcp-proxy はラップする MCP バックエンドの差し替えで任意の MCP に、mitm-proxy はポリシーの差し替えで任意の HTTP API に)。一方で、両者の粒度・許可表現でカバーできない領域も存在する。

git push の ref ACL がその典型で、ref / リポジトリ単位の判定は MCP ツール名でもホスト × HTTP メソッド × パスでも表現しきれない。このような領域は、**用途特化の自作カスタムプロキシを別レイヤとして用意する** ことで対処できる。本リポジトリでは git transport 用に `git-gateway` を自作している (Caddy + fcgiwrap + `git-http-backend` + bare リポジトリ + `pre-receive` フックの組み合わせ)。fetch はリバースプロキシで上流に流し、push は `pre-receive` フックで ref 許可リストによる受理判定を効かせる構造。

git push を粗粒度で扱う軽量代替として、mitm-proxy のアドオンで push を通す形 (= §3.1 の個別許可の例) も用意する。ただし push 内容に踏み込んだ ACL を効かせるなら、用途特化の専用プロキシ (= git-gateway) を立てる方が筋が良い。

詳細は [08-git-gateway.md](./08-git-gateway.md) と付録 [alt-git-mitm-proxy-addon.md](./appendix/alt-git-mitm-proxy-addon.md)。

## 4. レシピの評価軸

本リポジトリの全レシピは、以下の評価軸を満たすよう設計されている。読者が「自分のユースケースで採用可否」を判断する基準として使える。

1. **秘匿情報は作業コンテナ外に置く** — ホスト側 (`~/.config/devsbx/` など) に置き、プロキシ群だけが読み込む形にする (`env_file` / バインドマウント / 名前付きボリュームのいずれか)。作業コンテナのファイルシステムにも環境変数にも秘匿情報は入らない
2. **作業コンテナはプロキシのみと通信する** — Docker `internal: true` ネットワークで外部直接接続を遮断
3. **ACL はプロキシ側で評価する** — 作業コンテナ内で迂回できないこと (作業コンテナにプロキシ設定を上書きする手段を残さない)
4. **境界ドメインは信頼できる先に限定する** — 任意ホストへの外向き通信を許可しない

各レシピ章の冒頭で「この基準のどの境界条件に依拠しているか」を明示する。

## 5. 採用しない / 直交するアプローチ

本リポジトリで明示的に採用しなかった選択肢と、直交 / 補完的な他軸のアプローチを分けて整理する。

### 5.1 作業コンテナ内に秘匿情報を置いてスコープを絞る

秘匿情報自体が漏洩した時点でスコープの絞りは無効になる。**スコープは事後の被害範囲を絞るが、漏洩自体は防がない**。本リポジトリは漏洩経路を絶つ方向に振る。

### 5.2 直交 / 補完的な他軸のアプローチ

以下は本リポジトリと問題意識が部分的に重なる / 別軸で議論される選択肢。本リポジトリの主眼 (devcontainer 内の秘匿情報 + 外向き通信をプロキシ群に集約) とは性質が異なり、必要に応じて組み合わせる。

- **コンテナランタイム層の隔離** (rootless / seccomp / AppArmor / gVisor) — コンテナ侵害時の影響範囲を絞る方向。本リポジトリの「作業コンテナ内の秘匿情報 + 任意の外向き通信」課題とは別レイヤで、直交する。組み合わせ可能
- **L3/L4 層での外向き通信制御** (Tailscale / Cloudflare WARP / WireGuard 等) — 外向き通信の出口を別ネットワークに集約する。問題意識は近いが、L7 (HTTP メソッド / パス + MCP ツール名) 粒度の ACL や認証情報のプロキシ集約は別軸。組み合わせ可能
- **VM 隔離** (Docker Sandbox / GitHub Codespaces / Firecracker 等) — カーネル共有を捨てる方向。本リポジトリはカーネル共有 + Docker ネットワークで構成を軽く保つことが主眼で、追加の隔離レイヤとして組み合わせ可能。プロキシ群と作業コンテナの分離という上位構造を変えずに、隔離基盤だけ VM へ差し替える形でも適用できる

## 6. 適用範囲

本リポジトリは **個人開発者ローカルでの devcontainer 利用** を主眼とする。

- チーム共有環境 / CI / 本番環境への直接適用は想定外
- 2 つの基本コンポーネント (HTTP(S) ベース) でカバーする前提のため、**SSH / TCP-only / UDP は適用範囲外**

SSH 経由の git や DB 直接接続が必要な場合、L3/L4 プロキシを別軸として追加する方法はあるが、本リポジトリでは扱わない。

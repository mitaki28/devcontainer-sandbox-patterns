# 設計: 信頼境界をプロキシ群に置き、作業コンテナ自身は認可を制御できない

前章 ([01-problem.md](./01-problem.md)) では、秘匿情報自体の漏洩を防ぐ / 許可ドメイン内の意図しない操作を困難にする、という踏み込んだ隔離要件を整理した。これを 1 つの作業コンテナ内で実装しようとすると、条件が複数のレイヤに分散する。本章では、この分散を解消する設計原則と、本リポジトリのレシピが前提とするもの (信頼と保証) を示す。

## 1. 基本パターン

### 1.1 信頼境界をプロキシ群に集約し、作業コンテナ自身は認可を制御できない

一般的な AI エージェント向け devcontainer 構成例 (単一 Dockerfile + 任意で iptables / SNI ベース) は、**作業コンテナ内部に秘匿情報を置き、内部で制御する** 構造を取る。この構造で本書の踏み込んだ隔離要件 (秘匿情報自体の漏洩を防ぐ / 許可ドメイン内の意図しない操作を困難にする) まで組もうとすると、何を信頼すれば全体が成立するかが複数のレイヤに分散する (詳細は前章 [§2-§4](./01-problem.md))。

本リポジトリは逆に、**信頼境界** (= 何が崩れたら全体が崩れるかの線) **を独立した Docker コンテナ群 (= プロキシ群) に集約する** 形を取る。信頼境界が「プロキシ群の実装 + Docker ネットワーク設定」という限定された面に揃うため、何を信頼すれば全体が成立するかが明確化できる (具体的な内訳は §2):

- **プロキシ群だけが秘匿情報を読む** — API トークン / OAuth リフレッシュトークン / クラウドの認証情報はホスト側に置き、プロキシ群だけが読み込む。**作業コンテナのファイルシステムにも環境変数にも秘匿情報は入らない**
- **プロキシ群だけが認可判定する** (許可ホスト / 許可 ref / 許可 MCP ツール)
- **作業コンテナは「行動の意図」だけをプロキシ群に送る** (HTTP リクエスト / git コマンド / MCP 呼び出し)
- **作業コンテナから外への通信はすべてプロキシ経由で、それ以外の経路を Docker ネットワーク設定で塞ぐ** (`internal: true`)

### 1.2 本書における「認可」の定義

本書で「認可」と書くとき、これは **作業コンテナ外に影響を与える操作の許可** を指し、作業コンテナの内部で完結する操作は含まない。

たとえば、作業コンテナ内で AI エージェントが root 権限を持っていても、その操作が作業コンテナの内部に閉じている限り、作業コンテナ外から見れば影響はない。逆に、非 root ユーザーであっても、公開されているライブラリの main ブランチに push できる GitHub PAT を保持していれば、push 内容が CI の自動リリース等を経てライブラリの新バージョンとして公開され、外部 (利用者環境) に直接影響を与えうる。

作業コンテナ (AI エージェント) に与える権限を語るうえで本質的なのは、内部権限の有無ではなく **外部影響の有無** である。次節で扱う「権限を渡せない / 権限を増やせない」の 2 軸も、この外部影響を軸に考える。

### 1.3 「作業コンテナ自身は認可を制御できない」の意味

「認可を制御できない」というのは、具体的に次の 2 軸を意味する:

- **権限を渡せない** — 作業コンテナの中に API トークン / OAuth リフレッシュトークン / SSH 鍵を一切置かない。自身に与えられた認可を外部に渡す経路が無い
- **権限を増やせない** — 作業コンテナから外への通信は internal ネットワークで Docker ネットワーク設定により遮断され、プロキシ以外に出られない。自身に与えられた認可範囲を拡張する経路が無い

この 2 軸が揃うことで、万一作業コンテナ内の AI エージェント / シェル / プロセスが乗っ取られても、攻撃者が外部に持ち出せるのはプロキシが許可する範囲に閉じる。

### 1.4 操作許可のモデル

§1.1〜1.3 では骨格 (信頼境界をプロキシ群に置く + 作業コンテナ自身は認可を制御できない) を述べた。次に「外部への操作をどう許可するか」のモデルを示す。

出発点は **原則拒否 (default deny)** である。作業コンテナを Docker `internal: true` ネットワークに閉じ込めることで、作業コンテナから外部への通信路自体を Docker ネットワーク設定で消す ([03-foundation.md](./03-foundation.md))。この時点で、作業コンテナから外部に対するあらゆる操作 (HTTP リクエスト / MCP 呼び出し / git push / `npm install` / 任意 API 呼び出し) は何一つ通らない。

したがって、外部に対するすべての操作はプロキシ群を介して **「明示的な許可リストを書く」** ことでしか成立しない。本リポジトリでは、許可リストを書くための **基本コンポーネント** を 2 種類用意し、レシピごとに組み合わせる:

- **mcp-proxy** — 細粒度・**明示的** な操作許可。MCP ツール名で「この操作を許す」を直接書く
- **mitm-proxy** — 粗粒度・**暗黙的** な操作許可。ホスト × HTTP メソッド × パスで通信を許可し、その通信が表す操作 (`pnpm info <pkg>` / API GET / etc.) が暗黙的に許可される

両者は粒度と許可表現の方向性が異なるため、レシピごとに **どこを細粒度で書き、どこを粗粒度で済ませるか** を組み合わせる。具体的な対比と運用原則は §3 で扱う。

この 2 つでカバーできない領域 (例: git push の ref ACL は MCP ツール名でもホスト × HTTP メソッド × パスでも捌けない) には、**用途特化のカスタムプロキシを自作する** という手も取れる。本リポジトリでは git transport 用に `git-gateway` を自作している (§3.2)。

## 2. 本書の前提と保証

§1 の原則拒否 + 明示的な許可リストという構造は、何を信頼すれば全体が成立するかを限定された面に揃える。本節では、それを「何を信頼し (§2.1) ／どの攻撃者を想定し (§2.2) ／その下で何を守れるか (§2.3) ／保証の範囲をどう限定するか (§2.4)」の順に示す。

### 2.1 信頼するもの・しないもの

本書の設計は、次を信頼できる前提に立つ。いずれかが崩れれば保証も崩れる ＝ どこまでが本リポジトリの責任で、どこからが基盤・実装・上流の責任かを分ける線でもある。

信頼する技術スタックの 3 層 (脆弱性が無く、上流は提供者自身も信頼できるとみなす):

1. **基盤技術** — Docker engine / Docker Compose / Docker ネットワーク (`internal: true` による外部遮断)。土台のホスト側カーネル / Docker デーモンを含む
2. **プロキシ / ゲートウェイ実装** — `lib/mcp-proxy` / `lib/mitm-proxy` / `git-gateway`
3. **境界の上流** — ドメイン提供者 (github.com / atlassian.com / registry.npmjs.org / accounts.google.com 等) と、mcp-proxy でラップする MCP 実装 (公式 GitHub MCP / Atlassian MCP / Context7 / gcloud-mcp 等)。脆弱性が無いことに加え、提供者自身を信頼できること

加えてホスト側ユーザー (= 開発者本人) も信頼する (秘匿情報をホスト側に置く前提 (§4) はこの信頼に依拠する)。

一方で信頼しないのは作業コンテナの内部 (AI エージェント / 拡張 / 依存パッケージ) で、次節の通り内部では任意コード実行されうるものとして扱う。

プロキシ実装 (3 層の 2) について本リポジトリが取るアプローチは、自作領域を境界制御に必要な範囲まで絞り、外部依存を `@modelcontextprotocol/sdk` / mitmproxy / Caddy / `git-http-backend` のような標準コンポーネントに集約すること。プロキシ自身が信頼境界の中核を担う構造上、自前ロジックを増やすほど信頼すべき面が増えるため、各プロキシはこの方針を共通の設計指針とする (具体的な集約構造は [04](./04-mcp-proxy.md) §5、[05](./05-mitm-proxy.md)、[08](./08-git-gateway.md) を参照)。

### 2.2 想定する攻撃者

信頼しない領域 ＝ 作業コンテナの内部について、本書は最悪ケースを想定する:

- 作業コンテナ内の AI エージェント / 拡張 / 依存パッケージが任意コード実行能力を持つ
- 攻撃者は外部ネットワーク経由・サプライチェーン経由で作業コンテナに到達する (取得コンテンツ経由のプロンプトインジェクション、`npm install` した依存へのマルウェア混入など)

この攻撃者像を置くことが、§1 の設計が必要になる理由そのものである。作業コンテナの中でコードを実行されうる以上、中に秘匿情報を置けば読み出され、許可判定を中で完結させても作業コンテナ側から迂回される余地が残る。だから制御を作業コンテナの中で完結させず、境界をプロキシ群 (作業コンテナの外) に置く (§1.1 / §1.3)。逆に、カーネルや Docker デーモンを破る経路は想定に含めない (§2.1 で信頼する側に置いた) ＝ コンテナランタイム層やカーネルを破る攻撃は本書のスコープ外 (§6 のランタイム層の隔離は別レイヤ)。

### 2.3 その下で守れること

この前提 (§2.1) と攻撃者像 (§2.2) の下で、本書のレシピは次を **設計目標として置く**:

- 秘匿情報 (API トークン / OAuth リフレッシュトークン) が作業コンテナから漏洩しない
- 任意ホストへの外向き通信が通らない
- 許可外操作 (任意リポジトリへの push / 任意 MCP ツール呼び出し) が通らない

以降の章・付録からも、この「3 層を信頼できる前提の下で上記を満たすことを目標とする」形で参照する。

### 2.4 保証の範囲を限定する

§2.3 で立てた設計目標について、その範囲を限定する境界線を 3 つ明示する。

**上流組織が保持する情報の扱いは上流の運用に委ねる** — 3 層の条件 3 (境界の上流) の「上流を信頼できる」は、プロキシ群 → 上流の通信において、上流のウェブ解析やアクセスログに残るメタデータ (User-Agent / IP / 訪問パターン等)、および上流 API への引数として送信した内容 (検索クエリなど) の扱いを上流の運用に委ねることを含む。本リポジトリは外部に流出する経路の遮断は責任範囲とするが、これら上流側に保持される情報の扱いは上流提供者に委ねる。この前提を置きたくない用途には本リポジトリは適合しない。

**AI エージェント実行系 (Claude Code) 自身は本モデルの適用対象外** — 月額プラン (Claude.ai Pro / Max) 経由の利用では必須通信先ホストは公式に開示されているものの、認証フロー詳細や上流挙動の完全な仕様は公開されておらず、プロキシで通信を介入する運用が利用規約の解釈上グレー (公式に認可されていない自動化アクセスとみなされるリスク) になりうるため、本書は AI エージェント実行系自身を本モデルの対象外とする。構造的限界ではなく運用上の判断で、API トークン方式 (Amazon Bedrock / Google Vertex AI / Microsoft Foundry) 経由ではユーザー所有のクラウド認証情報で通信先もユーザー制御下に置けるため、原理的には本モデルを適用可能。

**このモデルを満たさないユースケースは対象外** — 「LLM に任意の URL を取得させたい」のような用途は、§2.1 の 3 層の条件 3 (境界の上流が限定される) が成立しない (= 任意ホストを許可した時点で境界が失われる) ため、本リポジトリの設計範囲外とする。任意ホスト fetch を扱うには、本書の前提とは別の保証の枠組み (Safe Browsing 等の事前チェック、LLM ベースの URL 判定など) が必要になる (詳細は付録 [incomplete-fetch-mcp.md](./appendix/incomplete-fetch-mcp.md))。

## 3. 基本コンポーネント: mcp-proxy と mitm-proxy

§1.4 で導入した 2 種類の基本コンポーネントを、許可単位と粒度の観点で具体的に対比する。

| 基本コンポーネント | 許可単位 | 粒度 | 操作許可の表現 | 主用途 |
|---|---|---|---|---|
| **mcp-proxy** | MCP ツール名 | 細 | 明示的 (「この操作を許す」を直接書く) | 認証 + 操作粒度を絞れるもの (GitHub, Atlassian, GCP, Context7 等) |
| **mitm-proxy** | ホスト × HTTP メソッド × パス | 粗 | 暗黙的 (通信が成立 = その通信が表す操作も成立) | 雑多な参照系 (`pnpm info`, レジストリ GET, ドキュメント参照等) |

両者は **直交する境界** (mcp-proxy は「何を MCP に頼むか」、mitm-proxy は「どこに HTTP を投げるか」) で、包含関係にはない。レシピごとに、許可したい操作の性質に応じて両方を組み合わせる。

### 3.1 原則: 粗粒度の mitm-proxy は読み取りのみ許可を主とし、書き込みは細粒度に集約

mitm-proxy は通信制御で操作を暗黙的に許す方式なため、ホストを 1 つ許可するだけでそのホスト上のサービスに対するあらゆる操作 (読み取り / 書き込み / 副作用のあるものも含む) が通る。粒度が粗いほど許可範囲が広く取られ、制御も難しくなるリスクが存在する。

一般的な粗粒度の外向き通信制御 (Squid / iptables) はホスト単位までで止まるが、本リポジトリの mitm-proxy はホスト × HTTP メソッド × パスまで絞れる。これにより、許可を読み取り操作のみに絞る運用 (= ドメインを許可しても更新系操作は通さない) が選択肢として成立する。

この表現力を活かすため、本リポジトリは以下の運用原則を置く:

- **外部への更新操作 (POST / PUT / PATCH / DELETE) は原則 mcp-proxy 軸を通す** — 操作粒度で明示的な許可リストを書けるため、何が許されているかが追跡しやすい
- **mitm-proxy でホストを許可するときは、原則として読み取りのみ許可 (参照系 GET / HEAD / OPTIONS のみ) を主として推奨する** — mitm-proxy は原則拒否 + 明示的な許可リストで動作する。許可を書く際の選択肢のうち、読み取り操作だけに絞れる区分を優先することで、暗黙的許可の影響範囲を読み取り系に限定する
- **書き込みが要る場合は明示的に個別許可する** — 信頼ホストの全メソッド許可、特定パス × メソッドの個別許可 (例: pnpm audit の副作用のない POST)、アドオンによる独自ロジック追加 (例: [`alternatives/git-mitm-proxy-addon/`](../alternatives/git-mitm-proxy-addon/) で git push を通す)、などの形を取る

この原則があると、2 つの基本コンポーネントが同じホストを扱う場面 (GitHub: MCP + git push / GCP: gcloud-mcp + SDK 直叩き) で重複が交通整理される。

具体的なポリシー構造 (trusted_hosts / readonly_hosts / allow_rules の 3 区分) や個別許可の書き方は [05-mitm-proxy.md](./05-mitm-proxy.md) §3.2 を参照。

### 3.2 基本コンポーネントでカバーできない領域: 用途特化のカスタムプロキシ

mcp-proxy / mitm-proxy は **汎用** の基本コンポーネントで、対象バックエンドを差し替えるだけで多様なユースケースに適用できる (mcp-proxy はラップする MCP バックエンドの差し替えで任意の MCP に、mitm-proxy はポリシーの差し替えで任意の HTTP API に)。一方で、両者の粒度・許可表現でカバーできない領域も存在する。

git push の ref ACL がその典型で、ref / リポジトリ単位の判定は MCP ツール名でもホスト × HTTP メソッド × パスでも表現しきれない。このような領域は、**用途特化の自作カスタムプロキシを別レイヤとして用意する** ことで対処できる。本リポジトリでは git transport 用に `git-gateway` を自作している (Caddy + fcgiwrap + `git-http-backend` + bare リポジトリ + `pre-receive` フックの組み合わせ)。fetch はリバースプロキシで上流に流し、push は `pre-receive` フックで ref 許可リストによる受理判定を効かせる構造。

git push を粗粒度で扱う軽量代替として、mitm-proxy のアドオンで push を通す形 (= §3.1 の個別許可の例) も用意する。ただし push 内容に踏み込んだ ACL を効かせるなら、用途特化の専用プロキシ (= git-gateway) を立てる方が筋が良い。

詳細は [08-git-gateway.md](./08-git-gateway.md) と付録 [alt-git-mitm-proxy-addon.md](./appendix/alt-git-mitm-proxy-addon.md)。

## 4. レシピの評価軸

本リポジトリの全レシピは、以下の評価軸を満たすよう設計されている。読者が「自分のユースケースで採用可否」を判断する基準として使える。

1. **秘匿情報は作業コンテナ外に置く** — ホスト側 (`~/.config/devsbx/` など) に置き、プロキシ群だけが読み込む形にする (`env_file` / バインドマウント / 名前付きボリュームのいずれか)。作業コンテナのファイルシステムにも環境変数にも秘匿情報は入らない
2. **作業コンテナはプロキシのみと通信する** — Docker `internal: true` ネットワークで外部直接接続を遮断
3. **ACL はプロキシ側で評価する** — 作業コンテナ内で迂回できないこと (作業コンテナにプロキシ設定を上書きする手段を残さない)
4. **境界ドメインは信頼できる先に限定する** — 任意ホストへの外向き通信を許可しない

各レシピ章の冒頭で「この基準のどの境界条件に依拠しているか」を明示する。

## 5. 別の視点: 作業コンテナに対する操作許可のフレームワーク

§1〜§4 で示した構造は、視点を変えれば **作業コンテナに対する操作許可のフレームワーク** として読める。各プロキシは「このプロトコルではこれらの操作が通る」を宣言する単位として働く。操作の許可とその実行に必要な秘匿情報の付与は表裏一体であり、本書の構造では両者をプロキシ単位でまとめて保持する。これにより、プロキシを追加・削除するだけで、作業コンテナに与える操作許可 (= 作業コンテナから外部に対してできること) を作業コンテナの外側で組み立て直せる構造を実現している、と捉えることもできる。

この視点に立つと、本書のレシピ集は「各操作を細粒度側に倒した構成」として読み替えられる。要件が異なれば各操作を粗粒度に倒した構成も同じパターンで記述でき、踏み込んだ隔離要件を満たす用途に限らず、作業コンテナに対する操作許可の設計や管理の枠組みとしても同じ構造を流用できる。以降の章を「どの操作をどの粒度で許可するか」の観点で読むと、こうした応用にも繋げやすい。

## 6. 組み合わせ可能なアプローチ

本書のレシピと組み合わせ可能 / 補完レイヤとして併用される他軸のアプローチ。

- **権限最小化** (PAT / OAuth スコープの最小化等) — 漏洩後の被害範囲を限定する。本書のレシピでも必要に応じて併用する (例: `cloud-mcp-with-short-lived-credential` では短寿命認証情報の発行と同時に権限スコープの最小化を行う)
- **コンテナランタイム層の隔離** (rootless / seccomp / AppArmor / gVisor) — コンテナ侵害時の影響範囲を絞る。本書が扱う秘匿情報の隔離 / 外向き通信の制御とは別レイヤ
- **L3/L4 層での外向き通信制御** (Tailscale / Cloudflare WARP / WireGuard 等) — 外向き通信の出口を別ネットワークに集約する。本書の L7 (HTTP メソッド / パス + MCP ツール名) 粒度の ACL や認証情報のプロキシ集約とは別軸
- **VM 隔離** (Docker Sandbox / GitHub Codespaces / Firecracker 等) — カーネル共有を捨てる。本書はカーネル共有 + Docker ネットワークで構成を軽く保つことを主眼とするが、プロキシ群と作業コンテナの分離という上位構造を変えずに隔離基盤だけ VM へ差し替える形でも適用できる

## 7. 適用範囲

本リポジトリは **個人開発者ローカルでの devcontainer 利用** を主眼とする。

- チーム共有環境 / CI / 本番環境への直接適用は想定外
- 本書のプロキシ群 (基本コンポーネント + 用途特化のカスタムプロキシ) が HTTP(S) ベースである前提のため、**SSH / TCP-only / UDP は適用範囲外**

SSH 経由の git や DB 直接接続が必要な場合、L3/L4 プロキシを別軸として追加する方法はあるが、本リポジトリでは扱わない。

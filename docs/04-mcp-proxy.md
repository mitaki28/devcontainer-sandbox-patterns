# 基本コンポーネント: mcp-proxy — 細粒度・明示的な操作許可

本章は本リポジトリで操作許可の記述を担う 2 つの基本コンポーネントのうち、**細粒度・明示的な操作許可** を担う `lib/mcp-proxy` の位置付けと機能を述べる。

## 1. 章のスコープ

`lib/mcp-proxy` は、作業コンテナ側のクライアント (Claude Code 等) とバックエンド MCP サーバの間に立つプロキシである。クライアントから見ると普通の streamable-HTTP MCP として振る舞い、その裏で 1 つのバックエンド MCP にリクエストを流す。

上流の MCP との間に立つことで、認証情報 (API トークン / OAuth トークン) をプロキシ側で持ってバックエンドに注入する、操作粒度をプロキシ側で絞る、といったことが可能になる。実装は TypeScript + Node + `@modelcontextprotocol/sdk`。具体的に何ができるかは次節の表を参照。

MCP プロトコルを基本コンポーネントの 1 つに選んだ理由は、操作許可との相性の良さにある。MCP はツール単位で意図が明確な操作をやりとりするため、意図した操作を細かい粒度で許可しやすい。実際、AI エージェント側の許可設定も MCP ツール単位が一般的で (例: Claude Code の `.claude/settings.json`)、mcp-proxy を挟むことで同じ粒度の許可設定をプロキシ側に移せる。加えて、MCP のエコシステムには GitHub / Atlassian / GCP 等のバックエンドが揃っており、mcp-proxy はトランスポート (stdio / streamable-HTTP) によらず中継できるので、既存の MCP をそのままプロキシ越しに使える。

## 2. 機能

| できること | 概要 |
|---|---|
| MCPの中継 | stdio / streamable-HTTP どちらのバックエンドでも streamable-HTTP として再公開 |
| 認証肩代わり | バックエンドへの Bearer / API キー注入、または OAuth 2.1 フロー駆動 (初回はブラウザで認可) |
| 操作粒度 ACL | MCP ツール名単位での許可 / 拒否。許可外はバックエンドに届かず、ツール一覧からも消える|
| 環境変数の絞り込み | プロキシの秘匿情報をバックエンド MCP に流さない (許可した環境変数だけ通す) |

## 3. 構成イメージ

mcp-proxy を実際に組み込んだ際の典型的な配置は次の通り。3 つの主体 (ホスト OS / プロキシコンテナ / 作業コンテナ) と秘匿情報の流れを示す:

![mcp-proxy を実際に組み込んだ際の典型的な配置](./mcp-proxy.excalidraw.png)

ポイント:

- **秘匿情報はプロキシだけが持つ** — ホスト側のファイル (`~/.config/devsbx/mcp-*.env` / `~/.cache/devsbx/mcp-<backend>/`) をプロキシだけが読み込み、作業コンテナには渡さない
- **認証情報はプロキシが注入** — API トークン / OAuth トークンはプロキシがバックエンドリクエストに注入する。作業コンテナからは見えない

## 4. 1 プロキシ 1 MCP の責務分離

本書の実装では 1 プロキシインスタンス = 1 バックエンド MCP の中継のみを行い、複数のバックエンド MCP の統合は行わない。

- コンテナごとに必要な認証情報を最小化できる (GitHub PAT は GitHub MCP のプロキシだけが持つ、等)
- MCP ごとにランタイム (Python / Node 等) やビルドが独立するので、互いに影響しない
- 1 つの MCP がクラッシュしたり OAuth 期限切れになっても、他の MCP には波及しない
- ログが「1 プロキシ = 1 MCP」で追えるので原因切り分けが楽

複数 MCP の統合は別のゲートウェイ層に任せ、プロキシ自体はバックエンドへの転送に絞る。

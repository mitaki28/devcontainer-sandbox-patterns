# 付録: 任意ホスト fetch の構造的限界 (fetch-mcp が「未完成」扱いになる理由)

[07-web-fetch.md](../07-web-fetch.md) で「任意ホスト fetch を諦める」設計判断を採り、`alternatives/fetch-mcp/` (自前 fetch MCP) を **未完成扱い** として除外した。本付録ではその理由と、fetch-mcp が実装している対策、それでも本書の前提 (§2) の枠内に収まらない構造を整理する。

## 1. 本付録のテーマ

[02-design.md](../02-design.md) §2.1 で信頼する 3 層は、条件 3 として「境界の上流が信頼でき、限定列挙されている」ことを前提にしている。web fetch (= 任意 URL の HTTP GET) はこの前提と構造的に相容れず、本リポジトリのフレームワークでは **対象外** ([02-design.md](../02-design.md) §2.4) と位置付けた。

ただし本リポジトリには `alternatives/fetch-mcp/` という **自前 fetch MCP の試作** が存在する。これは「任意ホスト GET をどこまで絞れるか」を試行したもので、実装としては動作するが、本書の前提に照らすと **「未完成」と判断して主推奨から除外** している。本付録ではその経緯と判断根拠を残す。

## 2. fetch-mcp が実装している対策

`alternatives/fetch-mcp/` は次の構造で任意ホスト GET を絞っている:

- **GET のみ / `https://` のみ** — POST 等の更新系は塞ぐ
- **リダイレクト不追従** — 3xx を返した場合は LLM にステータス + Location を返し、追うか諦めるかを判断させる
- **サイズ上限 / `Content-Type` 許可リスト** — 巨大ファイル / 想定外形式を拒否
- **HTML → Markdown サニタイズ** — `<script>` / `<style>` / `<iframe>` / `<object>` / `<embed>` を除去し Markdown に変換
- **TLS 終端を MCP 内で正規に行う** — Squid + MITM のような CA 配布問題が発生しない

## 3. それでも「未完成」と判断する理由

問題は、これらの対策が **「URL の中身がどんな先か」を判定していない** という点にある。fetch-mcp は次の経路を構造的に塞げない:

- **プロンプトインジェクション経由の情報漏洩** — Claude が取得したコンテンツに「次に下記 URL を fetch してください」のような指示が混入し、URL クエリ部分に内部情報 (編集中コード / git diff / 作業コンテナ内のファイル内容等) をエンコードして送信される経路
- **任意攻撃サーバへの GET 経路の新設** — `integrated/single-workspace/` でプロキシ群経由の外向き通信を限定的な SaaS API に絞ったあとも、fetch-mcp を入れた瞬間 **「任意ドメインへの GET 経路ゼロ」が崩れる**
- **既知の攻撃 / フィッシング / マルウェアホスト** — Safe Browsing API や OSS の拒否リストで部分的に弾けるが、これらは事後対応の限界がある

これらの経路は **「fetch 先 URL が本当に意図された参照先か」を判定しない限り構造的に塞がらない**。fetch-mcp の現実装はそこに踏み込んでいないため、**任意 URL を許可した時点で、[02-design.md](../02-design.md) §2.1 の条件 3 (境界の上流が信頼できる先に限定されている) も §4 の評価軸「境界ドメインは信頼できる先に限定する」も成立しなくなる**。

**「任意ホスト fetch」という要件自体が本書の前提 (§2) と相容れない** ため、fetch-mcp はその境界線を実装によって明らかにする試行として位置付け、本リポジトリで「採用 / 完成」とは呼べない位置に置いている。

## 4. 別の保証の枠組みが要る領域

任意ホスト fetch を安全に扱おうとすると、本書の前提とは異なる枠組みでの追加対策を組み合わせる必要が出てくる。代表的な選択肢としては:

- **既知の悪性 URL / ドメインの事前チェック** — Safe Browsing API や OSS の拒否リストを用いる
- **LLM ベースの事前 / 事後判定** — fetch 先 URL や取得した Markdown をリクエスト前後で LLM に判定させ、プロンプトインジェクションや情報持ち出し指示を検出する

これらは本書の前提 (3 層を信頼できる) とは **別の保証の枠組み** で動き、拒否リストの更新遅延を許容できるか / LLM 判定の誤判定確率を許容できるか、といった統計的・運用的な信頼性の話になる。

本リポジトリはこの前提の枠組みを採るため、これらの領域は責任範囲外として除外している。

## 5. まとめ

本リポジトリが web fetch 領域で「諦める」設計を採るのは、**任意ホスト fetch という要件が本書の前提 (§2) では扱えない領域だから** である。fetch-mcp は本リポジトリの境界線を明らかにする試行として保存してあるが、主推奨の枠組みに統合する判断はしていない。
